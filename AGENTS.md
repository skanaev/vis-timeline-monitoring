# AGENTS.md

## Назначение

Этот проект использует TypeScript + React 16.14 + Admiral DS + TanStack React Query v4.

Любой новый, изменяемый или рефакторимый код должен соответствовать правилам из этого документа.

Цель:
- писать понятный код
- сохранять строгую типизацию
- делать поведение компонентов предсказуемым
- упрощать поддержку и доработку проекта
- не плодить лишнюю сложность
- соблюдать единый стиль по всему проекту

Нельзя генерировать “быстрые костыли”, скрытые побочные эффекты и переусложненные абстракции.

## Общие правила работы с кодом

При работе с проектом всегда:

1. Сначала изучать существующий код вокруг.
2. Сохранять текущую архитектуру проекта, если она не сломана.
3. Предпочитать минимальные и безопасные изменения вместо ненужных переписываний.
4. Соблюдать стиль именования, структуру и подходы, уже используемые рядом.
5. Не делать лишний рефакторинг, если он не нужен для задачи.
6. Не добавлять новые библиотеки без явной необходимости.
7. Не менять бизнес-логику молча.
8. Если тип, ответ API или бизнес-правило непонятны — сначала делать выводы из существующего кода.
9. Если есть несколько способов реализовать задачу — выбирать самый простой, надежный и расширяемый.
10. Код должен быть таким, чтобы другой разработчик мог быстро его понять на ревью.

## Базовые принципы качества кода

### 1. Понятность важнее “умности”
Предпочитать простой, явный и скучный код.

Хорошо:
- небольшие функции
- понятные имена
- явные условия
- предсказуемый поток данных

Плохо:
- вложенные тернарники
- слишком много логики прямо в JSX
- магические числа и строки без имен
- побочные эффекты, спрятанные в хелперах
- абстракции ради абстракций

### 2. Строгая типизация обязательна
- Не использовать `any`, кроме совсем крайних случаев.
- Если тип неизвестен — использовать `unknown`, а затем сужать тип.
- Описывать сущности предметной области через `type` или `interface`.
- Предпочитать union-типы вместо произвольных строк.
- Не использовать безликие словари вида `Record<string, any>` там, где можно описать нормальную структуру.

### 3. Разделение ответственности
- UI-компоненты отвечают за отображение.
- Hooks отвечают за повторно используемую stateful-логику.
- API-слой отвечает только за HTTP-запросы.
- Мапперы/адаптеры преобразуют данные бэкенда в удобный для UI формат.
- Утилиты содержат чистые вспомогательные функции.

### 4. Предпочитать чистые функции
Все утилиты, селекторы и преобразования должны быть по возможности чистыми и детерминированными.

### 5. Не абстрагировать раньше времени
Не выносить хелпер, hook или компонент “на будущее”, если нет реального повторного использования или явной пользы.

## Правила по TypeScript

### Общие требования
- Всегда указывать типы там, где это улучшает понимание.
- Не усложнять типы без причины.
- Если тип слишком сложный — упростить структуру кода.
- Не строить чрезмерно “магические” generic-конструкции, если можно обойтись обычными типами.

### Предпочтения
- Для DTO, domain model и props использовать явно названные типы.
- Для union-типов использовать дискриминирующие поля, если это возможно.
- Для enum-подобных значений чаще использовать string union, а не `enum`, если нет строгой причины.
- Использовать `readonly`, если это помогает защититься от случайной мутации.
- Использовать `satisfies`, если это делает конфиги и мапы безопаснее и понятнее.

### Запрещено без веской причины
- `as any`
- двойные приведения типов вроде `as unknown as X`
- неочевидные generic-утилиты ради красоты
- незащищенный доступ к необязательным полям
- мутация объектов, пришедших извне

### Работа с nullable значениями
- Всегда явно обрабатывать `null` и `undefined`.
- Не полагаться на то, что “значение точно есть”, если это не гарантировано.
- Не злоупотреблять `!` non-null assertion.

## Правила по React

## Общие правила
- Использовать функциональные компоненты.
- Компоненты должны быть небольшими и сфокусированными.
- Один компонент — одна основная ответственность.
- Если JSX становится перегруженным — выносить части в дочерние компоненты или вычисления в переменные/хелперы.

### Структура компонента
Рекомендуемый порядок внутри файла компонента:
1. импорты
2. типы/интерфейсы
3. константы
4. компонент
5. локальные вспомогательные функции, если нужны

### Именование
- Компоненты: `PascalCase`
- hooks: `useSomething`
- функции-обработчики: `handleSomething`
- булевы значения: `is`, `has`, `can`, `should`
- константы: понятные именованные значения, а не магические литералы

### Состояние
- Хранить в state только то, что действительно должно там жить.
- Производные значения вычислять через селекторы, `useMemo` или обычные переменные, а не хранить отдельным состоянием.
- Не дублировать одно и то же состояние в нескольких местах.
- Не класть в state то, что можно получить из props или query-данных.

### Effects
- `useEffect` использовать только для реальных побочных эффектов.
- Не использовать `useEffect` для простых вычислений, которые можно сделать прямо при рендере.
- Любой `useEffect` должен иметь понятную цель.
- Следить за корректностью зависимостей.
- Не писать эффекты, которые сложно объяснить.

### useMemo / useCallback
Использовать только там, где есть реальная польза:
- стабилизация пропсов для тяжелых дочерних компонентов
- защита от лишних перерасчетов
- зависимости библиотек, чувствительных к ссылочному равенству

Не оборачивать все подряд в `useMemo` и `useCallback`.

### Обработчики
- Не писать сложную логику прямо внутри JSX.
- По возможности выносить в именованные функции.
- Обработчики должны быть короткими и понятными.

### Условия в JSX
- Не использовать сложные вложенные условия прямо в разметке.
- При сложной логике использовать ранний `return`, переменные или отдельные функции/компоненты.

### Списки
- Всегда использовать стабильные ключи.
- Не использовать индекс массива как `key`, если порядок элементов может меняться.

## Правила по React Query (@tanstack/react-query v4)

### Общие принципы
- React Query отвечает за серверное состояние.
- Не дублировать query-данные в локальный `useState`, если это не нужно для UI-редактирования.
- Ключи запросов должны быть стабильными, сериализуемыми и предсказуемыми.

### Query keys
- Использовать единый подход к ключам.
